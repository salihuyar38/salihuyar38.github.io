Building an Elastic Task Queue Simulation for HPC

July 2025

Introduction

High-Performance Computing (HPC) refers to the use of multiple computing resources working together to solve computational problems that are too large, complex, or time-consuming for a single machine.

Real-world HPC systems are significantly more complex than the project presented here. Production HPC environments often involve sophisticated schedulers, resource managers, fault tolerance mechanisms, monitoring systems, and execution across large-scale clusters or supercomputers.

The goal of this project was not to reproduce a production-level HPC system, but to explore some of the underlying concepts at a beginner-friendly level. In particular, I wanted to understand how workloads can be distributed dynamically, how failures affect running jobs, and how a scheduler can react to changing resource availability.

Project Overview

In this project, I implemented a small HPC-inspired elastic task queue simulation using Python, Bash, and Docker containers.

The system consists of one scheduler and two worker nodes. The Python application represents the workload executed by each worker. Workers receive a range of chunks to process and report their progress through log messages.

Docker Compose is used to create a lightweight cluster-like environment locally. A Bash scheduler script dispatches work, monitors node status, detects failures, and dynamically reassigns unfinished work when necessary.

Architecture

The system follows a simple master-worker architecture. The scheduler maintains a queue of work and distributes tasks to available worker nodes. When a worker fails, the scheduler detects the failure and migrates the unfinished work to a healthy node. 

                 Scheduler
                      |
                      v
              +---------------+
              |   Task Queue  |
              +---------------+
                 /         \
                /           \
               v             v

         hpc_node01     hpc_node02
          Chunks 1-3     Chunks 4-6

                             X
                       (Failure)

                             |
                             v

          hpc_node01 recovers
             Chunks 5-6

Although simplified, this architecture demonstrates concepts such as dynamic workload distribution, failure detection, and task migration, which are important topics in distributed computing and dynamic resource management.

What I Implemented

  • Python-based worker application
  • Chunk-based workload processing
  • Docker Compose cluster simulation
  • One scheduler container
  • Two worker containers
  • Dynamic task dispatching
  • Failure simulation using docker stop
  • Failure detection using Docker process monitoring
  • Automatic workload reassignment
  • Execution and recovery logging

Failure Experiment

The most interesting part of the project was the failure experiment. After assigning chunks 1-3 to hpc_node01 and chunks 4-6 to hpc_node02, the scheduler intentionally terminates hpc_node02 during execution.

The scheduler then checks the cluster state, detects that the worker is offline, identifies the unfinished chunks as lost work, and dynamically reassigns them to the surviving node.

While this project does not implement full fault tolerance, it helped me understand concepts that appear in real HPC systems such as dynamic resource management, workload migration, scheduler adaptation, and recovery after resource failures.

What I Learned

  • Basic scheduler design concepts
  • Dynamic workload distribution
  • Container-based cluster simulation
  • Failure detection and recovery strategies
  • Challenges of resource management in distributed systems
  • How HPC-inspired runtime systems adapt to changing resources
